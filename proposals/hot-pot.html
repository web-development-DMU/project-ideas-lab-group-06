Ideas — Hot Pot

  • Recipe sharing website
    • Website features
      • Create, edit, and publish recipes with ingredients and steps.
      • Rate/comment on recipes and save favorites.
      • Search by ingredient, cuisine, dietary tags.
    • The data of users:
      • id: Unique identifier for each user.
      • username: Unique name of the user.
      • email: Email address of the user.
      • created_at: Timestamp when the user account was created.
      • privileges: User privileges (e.g., admin, regular user).
    • The data of the recipes:
      • id: Unique identifier for each recipe.
      • title: Name of the recipe.
      • description: Brief overview of the recipe.
      • author_id: ID of the user who created the recipe.
      • servings: Number of servings the recipe makes.
      • prep_time: Time required for preparation (in minutes).
      • cook_time: Time required for cooking (in minutes).
      • difficulty: Difficulty level (e.g., Easy, Medium, Hard).
      • photo_url: URL to an image of the dish.
      • created_at: Timestamp when the recipe was created.
      • updated_at: Timestamp when the recipe was last updated.
    • Types of users
      • guest: Can view recipes but cannot create or edit them.
      • user: Can create, edit, and delete their own recipes.
      • admin: Has full access to all features and can manage users and content.
      • premium: Has access to premium features like advanced search and ad-free experience.
    • Routes and server actions
      • HTTP Status Codes:
        • 200 OK for a successful GET request
        • 303 See Other to redirect after successful POST (browser follows redirect as GET)
        • 400 Bad Request for validation error in POST request
        • 404 Not Found if requested resource does not exist
      • GET /recipes
        • What the server does: Retrieves a list of all recipes from the database.
        • What the client receives: JSON array of recipe objects including id, title, description, etc.
        • Steps the server performs:
          • Execute database query: SELECT id, title, description, author_id, servings, prep_time, cook_time, difficulty, photo_url, created_at, updated_at FROM recipes ORDER BY created_at DESC
          • Build the response: JSON array of the query results
          • Handle errors: If database error, return 500 Internal Server Error; if no recipes, return empty array with 200 OK
        • Example server action: For SQL query SELECT id, title FROM recipes WHERE author_id = ? (with parameter user_id), server fetches recipes by author and returns JSON like [{"id":1,"title":"Pasta"}]
      • POST /recipes
        • What the server does: Creates a new recipe in the database after validating input.
        • What the client receives: 303 See Other redirect to the new recipe's page.
        • Steps the server performs:
          • Validate incoming data: Check if title, description, ingredients are provided and valid; ensure user is authenticated.
          • Execute database queries: INSERT INTO recipes (title, description, author_id, ...) VALUES (?, ?, ?); then INSERT INTO recipe_ingredients for each ingredient.
          • Build the response: Redirect to GET /recipes/:new_id
          • Handle errors: If validation fails, return 400 Bad Request with error messages; if database error, return 500.
      • GET /recipes/:id
        • What the server does: Retrieves a specific recipe by ID.
        • What the client receives: List of the recipe or 404 if not found.
        • Steps:
          • Validate incoming data: Check if :id is a valid id.
          • Execute database query: SELECT * FROM recipes WHERE id = ?
          • Build response: Recipe data or 404
          • Handle errors: 404 if not found, 400 if invalid id
      • POST /staff/:staff_id/delete
        • What the server does: Removes a staff member from the database.
        • What the client receives: 303 redirect to the staff list page.
        • Steps:
          • Validate: Check staff_id exists, user authenticated.
          • Execute: DELETE FROM staff WHERE id = ?
          • Build response: Redirect to GET /staff page
          • Handle errors: 400 for invalid data, 404 if staff member not found
